# Sonde Security Audit — 2026-04-16

**Scope.** Structured white-hat sweep of the Sonde codebase: CLI auth,
Supabase RLS + RPCs, server (Hono) endpoints, UI client surface,
GitHub-Actions release pipeline, dependency CVEs. Verification done
against a fresh local Supabase with all migrations applied; no
production or staging systems touched.

**Method.** Three parallel recon passes produced ~40 candidate
findings. Each was then verified against end-state (not individual
migration files), with exploit reproduction for HIGH-severity claims.
Several recon "HIGH" items were stale and are called out in
§ Verified-not-live. Reusable artifacts are in
`scripts/security/` so this process is re-runnable.

---

## Executive summary

**The good.** Cross-program scoping for the core tables (experiments,
findings, directions, questions, artifacts, projects, activity_log,
managed sessions) is correctly RLS-gated via `user_programs()` /
`has_program_access()`. Self-escalation through `user_programs`
INSERT is blocked. Anon role is correctly denied on sensitive tables.
UI has a strong CSP + no `dangerouslySetInnerHTML` + proper
`safeAuthRedirect` hardening. No accidentally-committed secrets.

**The bad.** The initial pass found two HIGH database/RLS findings,
three MEDIUM hardening gaps, and one LOW-MEDIUM admin telemetry
exposure:

1. **HIGH** — `create_agent_token` lets any user who holds an
   `admin` role in *any* program mint agent JWTs scoped to *any*
   programs they request. Admin-of-alpha can create a working bot
   token for beta. Horizontal privilege escalation across research
   programs.
2. **HIGH** — `auth_events` is readable by every authenticated user
   and insertable by every authenticated user with arbitrary actor
   identity. Leaks login metadata across the org; attackers can
   forge audit events attributing actions to anyone.
3. **LOW-MEDIUM** — `db_size_snapshots` and its RPCs expose aggregate
   admin telemetry to non-admin authenticated users.
4. **MEDIUM** — CLI bot tokens (`sonde_bt_…`) embed the generated bot
   user's plaintext password in the token payload. Token theft should
   not remain useful after token expiry/revocation, but the password
   still broadens the blast radius while the token is live.
5. **MEDIUM** — `SONDE_CHAT_ALLOW_FRAME_AUTH=1` is a live env-flag auth
   bypass with no staging/production gate.
6. **MEDIUM** — `hono` in the server has seven moderate CVEs including
   one XSS and path-traversal patched in 4.12.14+.

**Current branch disposition.** This hardening branch fixes F-01,
F-02, F-05, F-06, and F-07; adds regression probes for the fixed RLS
paths; and confirms the PKCE concern was not live because the Supabase
Python client generated a PKCE challenge and falls back to its stored
verifier during `exchange_code_for_session`. F-03 is partially
mitigated by enforcing agent token expiry/revocation in the database
RLS helper and by making the CLI reject expired bot-token bundles
before password sign-in. The long-term cleanup remains replacing
password-bearing bot bundles with a server-side token exchange.

## Confirmed findings

Ordered by severity, then by effort to exploit.

### F-01: `create_agent_token` cross-program admin escalation   [HIGH]

**Where:** `supabase/migrations/20260329000010_auth_rbac.sql:244-313`
(function `create_agent_token`).

**Attacker capability required:** A valid login as a user who has
`role = 'admin'` in at least one program. Many Aeolus employees
likely qualify today.

**Blast radius:** Agent JWT with full read/write access to any
program(s) the attacker names in the RPC call. The token is valid
for up to 365 days and hits RLS the same way a legitimate agent
would. Effectively: an admin of any program can read or modify all
data in every other program, silently, via `sonde --token …`.

**Exploit (reproduced in `scripts/security/rls-exploit-tests.sql` —
TESTS G, H):**
```sql
-- Alice is admin of alpha only
SELECT create_agent_token('evil', ARRAY['beta','gamma'], 365);
-- Returns a signed JWT whose app_metadata.programs = ['beta','gamma']
```
Result (local repro): minted token for `['alpha', 'beta', 'gamma']`
despite alice's admin role existing only on alpha.

**Root cause:** The admin check collapses the per-program role to a
boolean:
```sql
SELECT bool_or(role = 'admin') INTO is_caller_admin
FROM user_programs WHERE user_id = caller_id;
```
It should require `role = 'admin' AND program = token_program` for
every program in `token_programs`.

**Likelihood (given attacker skill):** High if the attacker is a
trusted insider who goes rogue; low from external attackers (need
valid admin session first). Insider-threat blast radius makes it a
HIGH regardless.

**Suggested fix:** Change the admin check to
`token_programs <@ (SELECT array_agg(program) FROM user_programs WHERE user_id = caller_id AND role = 'admin')`,
so an admin can only mint for programs they actually administer.

### F-02: `auth_events` leaks org-wide login metadata; insertable fakes   [HIGH]

**Where:**
`supabase/migrations/20260402000006_create_auth_events.sql:23-28`.

**Policies:**
- `auth_events_select`: `FOR SELECT TO authenticated USING (true)`
- `auth_events_insert`: `FOR INSERT TO authenticated WITH CHECK (true)`

The migration comment states "admin guard is in the app layer" —
but the REST API (`/rest/v1/auth_events`) is reachable by any client
with an authenticated JWT. The "app layer" UI check is a button hide,
not a security boundary.

**Attacker capability required:** Any authenticated user with a
valid aeolus.earth session.

**Blast radius:**
- Read: every login event across all users and programs — email,
  `user_id`, program memberships, client version, timestamp, free-
  form `details` jsonb. Useful for targeting, timing correlation,
  org enumeration.
- Write: insert fabricated `login`/`logout`/`token_auth` events
  claiming to be **any actor** (including "human/ceo") with any
  `user_id`, any `programs`, any timestamp within the system clock's
  tolerance. Corrupts the forensic timeline; poisons admin-dashboard
  audit views.

**Exploit (TESTS A, B):**
```sql
-- As alice@aeolus.earth, scoped to program 'alpha':
SELECT * FROM auth_events;   -- reads bob's login metadata
INSERT INTO auth_events (event_type, actor, actor_email, user_id, programs)
VALUES ('login', 'human/ceo', 'ceo@aeolus.earth', '…', ARRAY['beta','alpha','gamma']);
-- succeeds — fake "CEO logged in to all programs" row persisted
```

**Likelihood:** Trivial once authenticated. Both exploits are a
single REST call away.

**Branch fix:** Direct INSERT is removed from authenticated users.
Auth-event writes now go through `record_auth_event()`, a
SECURITY DEFINER RPC that derives actor identity, user id, email,
and programs from the current JWT. Reads are scoped to admins of
overlapping event programs, with `shared` admins treated as org-wide
telemetry admins.

### F-03: CLI bot tokens embed plaintext account password   [MEDIUM]

**Where:** `cli/src/sonde/db/admin_tokens.py:98-104` (call site)
encodes the bundle via `cli/src/sonde/auth.py:703-707`
(`encode_bot_token`).

**What's in the token:** base64url-encoded JSON bundle containing
`{token_id, name, email, password, programs, …}`. `password` is
`generate_bot_password()` — 32 bytes of `secrets.token_urlsafe`, so
high entropy — but it is the **account's real password** for the
auto-created Supabase auth.users row, and it is transmitted and
stored inside the token.

**Attacker capability required:** Possession of a leaked or stolen
bot token string (CI secret, env var dump, error log, process-table
capture).

**Blast radius:** The attacker can:
1. Decode the token to recover `email` + `password`.
2. Sign into Supabase as that bot account indefinitely (the
   password doesn't rotate with the session).
3. Assume the bot's program scopes.

**Status:** Remediated by `20260417000001_opaque_agent_tokens.sql`
and the opaque-token exchange path. New agent tokens are `sonde_ak_…`
opaque secrets; the database stores only a SHA-256 hash. The hosted
`/auth/agent/exchange` endpoint uses the service-role key to call
`exchange_agent_token()`, which mints a short-lived JWT and re-checks
`agent_tokens.revoked_at` and `expires_at` on every exchange. The CLI
rejects legacy `sonde_bt_…` password-bundle tokens instead of using
Supabase password sign-in.

**Residual operational work:** rotate all existing `SONDE_*_CLI_AUDIT_TOKEN`
secrets and any developer/agent environment tokens, because legacy
tokens were hard-cut off by marking rows without `token_hash` revoked.

### F-04: CLI OAuth loopback PKCE concern   [VERIFIED NOT LIVE]

**Where:** `cli/src/sonde/auth.py:419-423`.

```python
params = CodeExchangeParams(
    auth_code=code,
    code_verifier="",
    redirect_to=redirect_url,
)
```

**Initial concern:** `code_verifier=""` looked like Sonde was
disabling PKCE in the loopback flow.

**Verification:** The Supabase Python auth client is configured with
`flow_type="pkce"` by the top-level Supabase client. Calling
`sign_in_with_oauth()` generates an OAuth URL with `code_challenge`
and `code_challenge_method`, and `exchange_code_for_session()` uses
the stored PKCE verifier when the passed `code_verifier` is empty.

**Branch action:** Added a regression test that asserts the generated
Google OAuth URL includes PKCE challenge parameters and a code comment
explaining the empty-string stored-verifier fallback. No security fix
was needed.

**Disposition:** Close as non-vulnerable unless the Supabase auth
client changes its PKCE storage behavior in a future dependency bump.

### F-05: `db_size_snapshots` readable to every authenticated user   [LOW-MEDIUM]

**Where:**
`supabase/migrations/20260402000003_db_size_metrics.sql:99-100`.

Policy: `snapshots_select FOR SELECT TO authenticated USING (true)`.

The table stores aggregate DB sizes, per-table byte counts, and
total artifact storage bytes. Not program-scoped — any logged-in
user can query it. Intended as an admin-dashboard data source.

**Attacker capability required:** Authenticated session.

**Blast radius:** Aggregate metrics — total DB size, per-table
sizes, storage total. Not research data, but useful for
reconnaissance of how busy a program is (bigger = more activity =
more interesting target).

**Likelihood:** Low impact alone; signal amplifier for targeted
attacks.

**Branch fix:** Gate snapshots and the DB-size RPCs behind
`public.is_sonde_admin()`, which this branch defines as admin of the
cross-cutting `shared` program rather than admin of any program.

### F-06: `SONDE_CHAT_ALLOW_FRAME_AUTH=1` is a production footgun   [MEDIUM]

**Where:** `server/src/app.ts:232-235`.

```ts
if ((process.env.SONDE_CHAT_ALLOW_FRAME_AUTH ?? "0").trim() === "1") {
  await next();     // bypass chat-session-token requirement
  return;
}
```

**Attacker capability required:** None if the flag is accidentally
set in production. Then anyone can open the chat WebSocket.

**Blast radius:** WebSocket auth bypass → direct access to the chat
streaming endpoint. The downstream handlers still check permissions
per message, but skipping the upgrade-time auth removes one
defense layer and likely lets anonymous clients open long-lived
connections (DoS / billing abuse path).

**Likelihood:** Non-zero because the flag has no "production"
short-circuit — an ops typo in Railway/Vercel config sets it
permanently.

**Suggested fix:** Honor the flag only when
`!isStrictEnvironment(env)` — the same pattern that guards the
device-auth dev-secret fallback. Alternatively, refuse to start if
the flag is truthy in staging/production.

### F-09: Release pipeline has no human gate from `staging` → production   [MEDIUM, architectural]

**Where:** `.github/workflows/promote-main.yml` →
`tag-on-promote.yml` → `release.yml`.

**Model:** A successful push to `staging` triggers `promote-main` →
auto-opens PR → auto-merges → auto-tags → auto-releases to GitHub
Releases. The only gate is whether 4 staging workflows (Staging
Smoke, Sync Staging Infra, Config Audit, CLI Hosted Audit) pass.

This is a deliberate choice, but it means any malicious code that
*passes smoke tests* reaches `@main` and then tagged release
without a human review step on the promotion PR. The smoke suite is
the entire security boundary for production.

**Suggested mitigations** (pick one, all are cheap):
- Gate auto-merge behind a 1-reviewer approval requirement.
- Add a manually-triggered `workflow_dispatch` for the production
  tag, separate from the staging-to-main promotion.
- Cosign / SLSA provenance on the release tarballs so end-users
  running `uv tool install --force "git+…@<tag>"` can verify. (Tags
  are mutable; anyone with push access to `main` can re-tag.)

**Status:** The selected mitigation is a required CODEOWNERS review
on `main`, with `promote-main` still opening/updating the staging-to-main
PR and enabling auto-merge. The machine can prepare the release, but a
human release owner must approve before branch protection lets the PR
merge.

## Verified-not-live

Candidates raised during recon that turned out to be already
patched or misread:

- `artifacts_all USING(true)` from `20260329000016_debug_artifacts_rls.sql`
  — superseded by `20260330000010_restore_artifact_scope.sql` (drops
  it) and further hardened by `20260330000016_repair_artifact_rls.sql`.
- `activity_log USING(true)` from `20260330000001_create_activity_log.sql`
  — tightened by `20260402000002_tighten_activity_log_rls.sql` to
  `can_access_record()`.
- `projects USING(true)` from `20260331000005_create_projects.sql`
  — tightened by `20260401000006_projects_rls.sql` to
  `program = ANY(user_programs())`.
- `managed_sessions / managed_session_cost_samples USING(true) TO authenticated`
  — hardened by `20260413000001_harden_managed_session_billing.sql`
  to `public.is_sonde_admin()` gates.
- `programs_read USING(true)` — intentional per migration comment
  ("programs are public metadata"). Programs table has no sensitive
  fields; confirmed.
- `user_programs` self-escalation via INSERT — RLS denies (TEST F).
- Anon role read of `auth_events` — RLS blocks correctly.
- CLI device-auth activation code brute-force — 31^8 ≈ 852B codes
  with unbiased random-byte selection (`device-auth.ts:274-291`)
  and 60-3600s TTL. Adequate entropy.
- `SONDE_DEVICE_AUTH_ENCRYPTION_KEY` fallback to `"sonde-dev-device-auth-secret"`
  — only in non-strict env; `assertEncryptionKey()` throws in
  staging/production when unset (`device-auth.ts:259-268`).
- UI `ReactMarkdown` XSS — no `rehype-raw`, URIs validated,
  `safeAuthRedirect` blocks `javascript:` / `data:`. Clean.
- UI Office-Online iframe — cross-origin iframe already browser-
  isolated; CSP `frame-src` whitelists only
  `view.officeapps.live.com` and `*.supabase.co`; no `postMessage`
  listener on the parent. Low residual.
- `.env` files in git (only `server/example.env`, a template) —
  false positive from the filename-pattern grep.
- No service-role key in UI bundle — confirmed.
- `merge-readiness.yml` `pull_request_target` — correctly scoped to
  `contents: read, pull-requests: read` with no PR-code checkout.

## Deferred / out of scope

- **Git history secrets scan.** This pass greps the working tree only.
  `gitleaks --log-opts='--all'` is a follow-up when a deeper baseline
  is needed.
- **Third-party GitHub Action SHA-pinning.** All actions are pinned
  to major tags (`@v6`, `@v8`). SHA-pinning would harden against tag
  force-push, but is operationally expensive to maintain.
  Dependabot handles tag updates today.
- **Branch-protection / tag-protection audit** — requires GitHub
  Admin UI access (not codebase state). User should confirm:
  `main` allows no force-push, PR-required; tags under `v*.*.*`
  cannot be deleted/re-pushed by default collaborators.
- **Deep RLS logic fuzzing** — the cross-program probe catches the
  common case (alice-sees-bob). It does not catch race conditions
  in `has_program_access()` under revoked agent tokens, or logic
  bugs in per-table policy qualifiers beyond what was observed.
- **Penetration testing against staging/production** — explicitly
  out of scope this pass.
